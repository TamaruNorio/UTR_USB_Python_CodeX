# 送信出力変更機能 要件定義

この文書は、UTR USB Pythonサンプルへ送信出力変更機能を追加する前の要件定義である。

現時点では、送信出力変更の実装はまだ行わない。  
まず、読み取り専用で確認できる現在状態を前提に、今後どの範囲まで変更機能を実装するかを安全に整理する。

## 1. 背景

現場確認では、Inventory結果を評価するときに、現在の送信出力設定を確認したい場面がある。

すでに対応済みの範囲として、Inventory前に以下を読み取り表示する機能がある。

- 送信出力設定の読み取り
- 周波数設定の読み取り
- raw hexの表示
- 読み取り失敗時もInventoryを継続する処理

次の改善候補として、送信出力を一時的に変更して、読み取り距離・RSSI・アンテナごとの差を確認できるようにする要望がある。

ただし、送信出力変更は、現場影響、機器設定、電波法、設定復元、FLASH保存有無に関係するため、読み取り機能より慎重に扱う。

## 2. 目的

送信出力変更機能の目的は、実機検証時に以下を安全に確認できるようにすることである。

- 現在の送信出力を確認する。
- ユーザーが明示的に選択した場合だけ送信出力を一時変更する。
- 変更後の送信出力を再読み取りして確認する。
- Inventory結果、RSSI、アンテナごとの読み取り傾向を確認する。
- 必要に応じて元の送信出力へ戻せるようにする。

## 3. 非目的

初期実装では、以下を行わない。

- FLASHへの自動保存
- 起動時デフォルト設定の変更
- 周波数設定の変更
- 8CHアンテナ切替制御の実装
- 8CH機での未確認動作の有効化
- 送信出力範囲外値の送信
- ユーザー確認なしの送信出力変更
- `UHF_SET_INVENTORY_PARAM` の自動送信

## 4. 安全原則

### 4.1 明示確認の原則

送信出力変更は、ユーザーが明示的に `y` を入力した場合だけ実行する。

デフォルトは変更しない。

```text
送信出力を一時変更しますか？ [y/N]:
```

### 4.2 読み取り優先の原則

変更前に、必ず現在の送信出力を読み取る。

読み取りに失敗した場合、初期実装では変更処理へ進まない。

理由:

- 元の値が分からないと復元できない。
- 現場で現在状態を説明できない。
- 誤った変更後状態を残すリスクがある。

### 4.3 一時変更優先の原則

初期実装では、FLASH保存を行わない。

変更は検証中の一時変更として扱う。

FLASH保存が必要な場合は、別PR・別設計とする。

### 4.4 復元可能性の原則

送信出力変更後は、終了時に元の送信出力へ戻す設計を基本とする。

ただし、仕様上または実機挙動上、元の値へ戻せない可能性がある場合は、実装前に確認する。

### 4.5 失敗時継続の原則

送信出力変更に失敗した場合、原則としてInventoryは中止する。

理由:

- 意図した送信出力で検証できない。
- 変更途中の状態が不明になる。
- 読み取り結果の解釈を誤る可能性がある。

ただし、送信出力「読み取り」だけに失敗した場合は、現在の実装どおりInventory継続を許可できる。  
「変更」失敗と「読み取り」失敗は扱いを分ける。

## 5. 対象機種

初期実装の主対象は、実機確認済みの UTR-SUN02-4CH / USM02 とする。

| ROMシリーズ | 機種名 | 初期方針 |
|---|---|---|
| USM02 | UTR-SUN02-4CH | 実装候補 |
| USM06 | UTR-SUN02V-8CH | 初期実装対象外 |
| USM08 | UTR-SUN02-8CH | 初期実装対象外 |
| USM01 | UTR-S201 | 初期実装対象外 |
| USM05 | UTR-SHR201 | 初期実装対象外 |
| unknown | 不明機種 | 対象外 |

8CH機はアンテナ体系や運用条件が異なるため、送信出力変更機能も4CHと同一扱いしない。

## 6. 操作フロー案

初期実装では、以下の流れを想定する。

```text
1. COMポート選択
2. 接続
3. ROMバージョン読み取り
4. ROMシリーズ判定
5. コマンドモード設定
6. Inventory前のリーダライタ設定読み取り
   - 送信出力
   - 周波数設定
7. 送信出力を一時変更するか確認
8. 変更しない場合
   - そのままInventoryへ進む
9. 変更する場合
   - 現在値を保存
   - 入力値を検証
   - 一時変更コマンド送信
   - 変更後の値を再読み取り
   - 成功した場合だけInventoryへ進む
10. Inventory実行
11. 終了時に元の送信出力へ復元
12. 復元後の値を再読み取り
13. アンテナ設定復元
14. 接続終了
```

## 7. 入力仕様案

### 7.1 入力形式

送信出力値は、ユーザーが理解しやすい dBm 表記で入力する。

例:

```text
送信出力値を入力してください（例: 24.0、変更中止は q）:
```

### 7.2 許可範囲

許可範囲は仕様書と実機確認に基づいて定義する。

現時点で範囲が確定していない場合、コードへ推測値を入れない。

範囲が確定するまでは、要件上は以下とする。

```text
許可範囲: 要確認
刻み幅: 要確認
機種差分: 要確認
```

### 7.3 入力バリデーション

最低限、以下を確認する。

- 数値として解釈できること。
- 許可範囲内であること。
- 許可刻みであること。
- `q` で中止できること。
- 空Enterでは変更しないこと。

## 8. 表示仕様案

変更前後で、以下を表示する。

```text
=== 送信出力一時変更 ===
現在の送信出力: xx.x dBm
FLASHは変更しません。
送信出力を一時変更しますか？ [y/N]: y
変更後の送信出力を入力してください: xx.x
送信出力を一時変更しました。
変更後の送信出力: xx.x dBm
```

終了時の復元表示例。

```text
送信出力を元に戻します。
復元対象: xx.x dBm
FLASHは変更していません。
送信出力を元に戻しました。
復元後の送信出力: xx.x dBm
```

失敗時の表示例。

```text
送信出力の変更に失敗しました。
Inventoryは開始しません。
現在の設定を再確認してください。
```

## 9. エラー処理要件

| 発生箇所 | 方針 |
|---|---|
| 変更前読み取り失敗 | 変更処理へ進まない。Inventoryはユーザー確認後に継続可とするか検討 |
| 入力値不正 | 再入力させる |
| 範囲外 | 送信しない |
| 書き込みNACK | Inventoryを開始しない |
| 書き込み後読み取り失敗 | Inventoryを開始しない、または警告して中止 |
| Inventory中Ctrl+C | 送信出力復元を試みる |
| 復元失敗 | 明確に警告し、手動確認を促す |
| シリアル切断 | 復元できない可能性を警告する |

## 10. 復元要件

送信出力を変更した場合は、以下を保持する。

- 変更前の送信出力値
- 変更後の送信出力値
- 変更実行結果
- 復元要否

終了時には、以下の順序で復元を試みる。

```text
1. Inventoryを終了する
2. 受信バッファを可能な範囲でクリアする
3. 送信出力を元に戻す
4. 復元後の送信出力を読み取る
5. コマンドモード用アンテナ設定を元に戻す
6. 接続を閉じる
```

アンテナ設定復元との順序は実機確認で検証する。  
現時点の仮案では、Inventoryに影響する送信出力を先に復元し、その後アンテナ設定を復元する。

## 11. コマンド実装要件

送信出力変更コマンドを実装する場合、以下を満たす。

- コマンド生成は純粋関数にする。
- dBm値から送信用バイト列への変換を純粋関数にする。
- 許可範囲外の場合は例外またはエラーを返し、送信しない。
- レスポンス解析も純粋関数にする。
- NACK時の表示を既存のNACK処理に合わせる。
- FLASH保存を行うコマンドとは分離する。

## 12. テスト要件

実機通信なしで、以下をpytestで確認する。

- dBm値から送信用値への変換
- 送信用値からdBm値への変換
- 範囲外値を拒否すること
- 不正入力を拒否すること
- 変更コマンド生成
- 変更レスポンス解析
- NACK時の扱い
- 変更前読み取り失敗時に変更へ進まないこと
- 書き込み失敗時にInventoryへ進まないこと
- 復元対象値を保持できること

実機テストは別途行う。

## 13. 実機確認要件

初期実装後、以下を確認する。

| 確認項目 | 内容 |
|---|---|
| 変更前読み取り | 現在の送信出力が表示される |
| 一時変更 | 指定した値へ変更される |
| 再読み取り | 変更後の値が表示される |
| Inventory | 変更後出力で読める |
| RSSI傾向 | 出力変更に応じて変化が見えるか確認 |
| 終了時復元 | 元の値へ戻る |
| Ctrl+C時復元 | 中断時も元の値へ戻る |
| 失敗時挙動 | NACKや切断時に危険な継続をしない |

実機ログは、`docs/real_device_verification_template.md` の形式で記録する。

## 14. GitHub運用要件

送信出力変更機能は、最低でも以下のPRに分ける。

### PR 1: 要件定義

- この文書を作成する。
- コード変更なし。

### PR 2: 純粋関数とテスト

- 送信出力値の変換関数
- コマンド生成関数
- レスポンス解析関数
- pytest追加
- 実機通信なし

### PR 3: UI接続

- 実行時の入力画面追加
- 変更前確認
- 変更後確認
- エラー時中止
- 復元処理の枠組み

### PR 4: 実機確認後の調整

- 表示文言の改善
- 実機ログに基づく補正
- ドキュメント更新

## 15. 実装前に確認すべき未解決事項

以下は、実装前に仕様書または実機ログで確認する。

- 送信出力設定コマンドの正確なデータ形式
- dBm値とバイト列の変換式
- 許可される送信出力範囲
- 設定刻み幅
- RAM一時変更とFLASH保存の区別
- 機種別の送信出力範囲差分
- アンテナ個別送信出力設定との関係
- 8CH機での扱い
- 復元順序
- 変更失敗時の安全な終了手順

## 16. 初期実装の完了条件

送信出力一時変更機能を実装する場合、初期完了条件は以下。

- UTR-SUN02-4CH / USM02 のみ対象。
- 変更前の値を読み取れる。
- ユーザー確認後のみ変更する。
- 範囲外値を送信しない。
- FLASH保存を行わない。
- 変更後の値を再読み取りできる。
- Inventoryを実行できる。
- 終了時に元の値へ戻せる。
- Ctrl+C時も復元を試みる。
- 復元失敗時は明確に警告する。
- pytest、dev_check、git_preflightが成功する。
- 実機確認ログがマスク済みで残っている。

## 17. 判断

現時点では、送信出力変更機能はまだ実装しない。

次の妥当なステップは以下。

1. 送信出力変更コマンド仕様の確認
2. 許可範囲と刻み幅の確認
3. RAM一時変更とFLASH保存の違いの確認
4. 純粋関数だけのPR作成
5. 実機通信へ接続するPR作成

安全性を優先し、最初のコード実装は「一時変更のみ」「USM02のみ」「FLASH非変更」に限定する。
