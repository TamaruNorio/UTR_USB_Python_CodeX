# RSSI Validation Plan

## 目的

RSSI 検証の目的は、Python 表示値と UTRRWManager 表示値の差を確認することです。

この段階では、補正式を先に決めません。複数条件の比較結果と仕様上の根拠を確認してから、コード変更が必要かを判断します。

## 現在の実装方針

- RSSI を符号付き 16bit として扱う。
- 値を 10 で割って表示用の値にする。
- 根拠なく補正しない。

## 検証時に記録する項目

| 項目 | 記録内容 |
|---|---|
| 日付 | 測定日 |
| 機種 | UTR-S201/202 など |
| 接続方式 | USB シリアルなど |
| COM ポート | `COMx` のように記録 |
| 通信速度 | 例: `115200` |
| アンテナ | 使用アンテナをマスク可能な範囲で記録 |
| タグ種別 | タグ種別や形状 |
| 距離 | アンテナとタグの距離 |
| Python 表示 RSSI | Python 側の表示値 |
| UTRRWManager 表示 RSSI | UTRRWManager 側の表示値 |
| 差分 | Python 表示値と UTRRWManager 表示値の差 |
| 備考 | 条件差、再測定メモなど |

## 判断基準

次の観点で差分を確認します。

- 差が一定オフセットか。
- 距離やタグで変動するか。
- 符号、エンディアン、単位の問題か。
- 表示丸めの問題か。

## 禁止事項

- 1 回の測定だけで補正式を入れない。
- 顧客現場ログや実タグ ID を未マスクで commit しない。
- 実シリアル番号、顧客情報、実 IP、未マスク実測ログを commit しない。
- UTRRWManager との差が見えただけで、根拠なく RSSI の補正式を決めない。

## 次にコード変更が必要になる条件

次のどちらかを満たす場合に、コード変更を検討します。

- 複数条件で一貫した差分が確認された場合。
- 仕様書上の根拠が確認できた場合。

コード変更を行う場合も、差分の根拠、検証条件、期待値をドキュメントまたはテストに残してから進めます。
